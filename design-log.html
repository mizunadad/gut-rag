// 01 — なぜベクトル化しないか

構造的継承（lineage retrieval）を選んだ理由

「腸で感じた直感を、記録し、問いを通じて言語化する」
個人知の蓄積。継承されるのは知識文書という思想の枠組みのみ。
身体知が先行する、という事実
五蘊の「受」（感受）は「想」（表象）より先に来る。
メルロ＝ポンティの「身体と世界の接触」は、意味が生まれる前の出来事。
島皮質が「これだ」と感じる瞬間は、言語の手前にある。

ベクトル検索は、言語化された意味の空間を泳ぐ道具。身体知は、その手前にある。
「似ているものを探す」ではなく「この筋から来た言葉を渡す」
般若心経→哲学→神経科学→身体知、という線は、類似度ではなく継承の順序。
ベクトル検索を使うとこの筋が切れる可能性がある。
ベクトル化への問い：ベクトルが「ざわっと」に最も近いと判定したものが、哲学でも般若心経でもなかったとき、何を失うか。
もしベクトル化したら
＋ 「会議中のざわっと」と過去の似た記録が意味的に照合される
＋ 予期しない接続が生まれる
− 思想の筋が切れる可能性がある
− 「なぜこれが返ってきたか」が不透明になる
− バックエンドが必要になる

次の地平：完全構造図のrequires/enablesをそのまま辿る「グラフ探索型RAG」が、思想の筋を最も忠実に継承できる可能性がある。

// 06 — 体験モードの設計哲学

摩擦を体験の前に置かない

「APIキーを設定しないと動かない」は、まだ体験していない人に向けたハードルだ。
体験してから設定する、の順が正しい。
設計の判断
APIキー未設定の状態で「もっと深く探る」を押した瞬間、デモ応答が流れる。
「✦ 体験モード」バッジと、APIキー設定への自然な導線を同時に表示。

24件のプリセット： 4カテゴリ × 6チョイス。
プリセットは「典型的な気づき」であり、パーソナライズ応答との差分がAPIキー設定への動機になる。
ことばの種（plain_insight + resonance_question）
Socratic応答後に折りたたみセクションとして出現。タップで開くと：
・plain_insight：宗教・学術フレームを外した体感の言葉（1〜3文）
・resonance_question：知識から記録へ向かう問い（gut-loggerへの橋）

設計原則：「ことばの種」は解説ではなく、「開く扉」。
「この言葉、なんだろう」と思わせれば成功。説明しきらないことが価値。

現状：lineage 4件のみ plain_insight を追加済み。残27件は未追加。
// 07 — sw.js の教訓

サービスワーカーが外部オリジンを横取りしていた

// BUG: TypeError: body is already used
旧 sw.js は外部オリジン（api.anthropic.com）の fetch も横取りしていた。
レスポンスを r.clone() + return r で二重消費 → API 呼び出しが無音で失敗。
修正（v6）
// 外部オリジンは即座にバイパス
if (url.origin !== self.location.origin) return;
この1行をfetchイベントの先頭に入れる。
GitHub Pages + Anthropic API の構成では必須。

教訓：PWAのサービスワーカーは、外部API通信と共存する場合にオリジンチェックが不可欠。
外部オリジンをバイパスしないと、APIレスポンスのbodyが消費済みになりエラーになる。

// 09 — 造語・新概念

このプロジェクトで生まれた言葉

lineage retrieval（リネージ・リトリーバル）
定義：思想の継承の連鎖（lineage）に沿って文書を取得するRAGの手法。クエリとの類似度ではなく、設計者が定めた知的系譜の構造に基づいて文書を選択・注入する。

既存手法との違い：
・vector similarity search：クエリとの意味的近さで取得
・keyword retrieval：キーワード一致で取得
・lineage retrieval：知的継承の連鎖の各層を代表する文書を取得

このプロジェクトでの実装：
般若心経（hannya-001）→ 哲学（tetsugaku-001）→ 神経科学（nou-007）→ 身体知（kotoba-011）の4層から必ず1件ずつ取得し、Socraticのプロンプトに注入する。

なぜこの名前か：
身体知は言語化の「前」に存在するため、意味の空間（ベクトル空間）での類似検索とは構造的に不一致。思想の筋は類似度ではなく継承の順序であり、その継承（lineage）に沿って取得することが設計思想と一致する。
// 10 — Zenn記事の訴求点

この経験から伝えたいこと

①
「なんか違う」を無視してきた
入口は共感。品質エンジニアとして、あるいは人間として、言語化できない違和感を脇に置いてきた経験。
②
なぜベクトル化しなかったか
身体知は言語化の前にある。だからベクトル空間（意味の空間）での類似検索は、構造的に一致しない。
③
このツールが解くのは何か（再定義）
個人の直感を言語化して理解に繋げること。集合知ではなく個人知。過剰な主張を外し、今できることに誠実に。
④
もしベクトル化したら何が得られ、何が失われるか
読者の探求心を刺激する。「自分だったらどう設計するか」を考えさせる。
⑤
制約が思想を生んだ
「バックエンドなし・vanilla JS・GitHub Pages完結」という制約が、「なぜベクトル化しないか」という問いを産んだ。やってみた価値は、こうやって紡がれる。
