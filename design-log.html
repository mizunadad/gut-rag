// 01 — なぜベクトル化しないか

構造的継承（lineage retrieval）を選んだ理由

「腸で感じた直感を、記録し、問いを通じて言語化する」
集合知ではなく個人知の蓄積。継承されるのは知識文書という思想の枠組みのみ。
身体知が先行する、という事実
五蘊の「受」（感受）は「想」（表象）より先に来る。
メルロ＝ポンティの「身体と世界の接触」は、意味が生まれる前の出来事。
島皮質が「これだ」と感じる瞬間は、言語の手前にある。

ベクトル検索は、言語化された意味の空間を泳ぐ道具。身体知は、その手前にある。
「似ているものを探す」ではなく「この筋から来た言葉を渡す」
般若心経→哲学→神経科学→身体知、という線は、類似度ではなく継承の順序。
ベクトル検索を使うとこの筋が切れる可能性がある。
ベクトル化への問い：ベクトルが「ざわっと」に最も近いと判定したものが、哲学でも般若心経でもなかったとき、何を失うか。
もしベクトル化したら
＋ 「会議中のざわっと」と過去の似た記録が意味的に照合される
＋ 予期しない接続が生まれる
− 思想の筋が切れる可能性がある
− 「なぜこれが返ってきたか」が不透明になる
− バックエンドが必要になる

次の地平：完全構造図のrequires/enablesをそのまま辿る「グラフ探索型RAG」が、思想の筋を最も忠実に継承できる可能性がある。
// 02 — 2つの流れの関係性

経験文書と知識文書は今どこで出会うか

「並列注入であり、合流はLLMに委ねている」が正確な現状。
経験文書（gut-logger）
集計値のみを注入：当たり率・よく感じる文脈・meaningタグの傾向
自由入力テキスト（note）は一切送信しない
→ LLMは「この人のパターン」を知る
知識文書（knowledge.js）
31件の文書から思想の筋を代表する文書を選択・注入
→ LLMは「このプロジェクトの文脈と語彙」を知る
2つの流れの現在地
検索レベルでは相互に無関係。LLMの中で初めて出会う。
未実装の接続：meaningタグ「場の空気を感じていた」がkotoba-003（気配を読む）に対応するような紐づけはまだない。
これが実装されると、経験と知識が検索レベルで繋がる。

// 05 — Zenn記事の訴求点

この経験から伝えたいこと

①
なぜベクトル化しなかったか
身体知は言語化の前にある。だからベクトル空間（意味の空間）での類似検索は、構造的に一致しない。
②
このツールが解くのは何か（再定義）
集合知ではなく、個人の直感を言語化して理解に繋げること。過剰な主張を外し、今できることに誠実に。
③
もしベクトル化したら何が得られ、何が失われるか
読者の探求心を刺激する。「自分だったらどう設計するか」を考えさせる。
④
次の地平：グラフ探索型RAG
完全構造図のrequires/enablesをそのまま辿る実装。思想の筋を最も忠実に継承できる可能性。続きを読みたくなる終わり方。
⑤
制約が思想を生んだ
「バックエンドなし・vanilla JS・GitHub Pages完結」という制約が、「なぜベクトル化しないか」という問いを産んだ。やってみた価値は、こうやって紡がれる。

// 07 — 造語・新概念

このプロジェクトで生まれた言葉

lineage retrieval（リネージ・リトリーバル）
定義：思想の継承の連鎖（lineage）に沿って文書を取得するRAGの手法。クエリとの類似度ではなく、設計者が定めた知的系譜の構造に基づいて文書を選択・注入する。

既存手法との違い：
・vector similarity search：クエリとの意味的近さで取得
・keyword retrieval：キーワード一致で取得
・lineage retrieval：知的継承の連鎖の各層を代表する文書を取得

このプロジェクトでの実装：
般若心経（hannya-001）→ 哲学（tetsugaku-001）→ 神経科学（nou-007）→ 身体知（kotoba-011）の4層から必ず1件ずつ取得し、Socraticのプロンプトに注入する。

なぜこの名前か：
身体知は言語化の「前」に存在するため、意味の空間（ベクトル空間）での類似検索とは構造的に不一致。思想の筋は類似度ではなく継承の順序であり、その継承（lineage）に沿って取得することが設計思想と一致する。既存の技術用語では表現できないため、このプロジェクトで命名した。
