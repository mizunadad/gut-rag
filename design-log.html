ADR-001 採択済み

なぜベクトル検索を使わないか

身体知は、言語化される以前の状態で存在している。
ベクトル検索は「すでに言語化されたもの」同士を比べる道具といえる。言語化以前の感覚にそれを当てることは、構造的に一致しない。
// CONTEXT — 背景
五蘊の「受」（感受）は「想」（表象）より先に来る。メルロ＝ポンティの身体論では、世界との接触は意味が生まれる前の出来事として記述される。島皮質が「これだ」と感じる瞬間は、言語の手前にある。

ベクトル検索は、すでに言語化された意味の空間を泳ぐことに最適化された道具といえる。言語化以前の感覚に対して意味的に近い文書を当てることは、構造的に一致しない。
// DECISION — 決定
lineage retrieval（系譜的取得）を採用している。

これは、現代のAI開発でいう「ガードレール設計」に相当する考え方といえる。身体知マップ・完全構造図があらかじめガードレールとして機能し、感覚ラベル・meaningタグがどの知識文書に対応するかをその上に定義している。

LLMはこのガードレールの外に出ない。意味の空間を自由に泳ぐのではなく、ガードレール上を辿って文書を取得する。ベクトル検索が起きないのは制約ではなく、この設計の帰結といえる。

マッピングを手で書くということは、実践者の判断そのものを設計の中心に置くということ。その判断の集積が、このシステムの知性を形成している。
// CONSEQUENCES — 帰結
+思想の筋（般若心経→哲学→神経科学→身体知）が応答に流れ続ける
+「なぜこの文書が使われたか」が完全に透明で追跡可能
+バックエンド不要。GitHub Pages完結が可能
−過去の類似記録との意味的照合はできない
−新しい感覚ラベルやmeaningタグの追加には手動マッピングが必要
lineage retrieval とは何か
定義：思想の継承の連鎖（lineage）に沿って文書を取得するRAGの手法。クエリとの類似度ではなく、設計者が定めた知的系譜の構造に基づいて文書を選択・注入する。

既存手法との対比：
· vector similarity search：クエリとの意味的近さで取得
· keyword retrieval：キーワード一致で取得
· lineage retrieval：知的継承の連鎖の各層を代表する文書を取得

このプロジェクトでの実装：
hannya-001（般若心経）→ tetsugaku-001（哲学）→ nou-007（神経科学）→ kotoba-011（身体知）の4層から必ず1件ずつ取得し、Socraticのプロンプトに注入する。
ADR-002 採択済み

なぜLLMを「問う役」にしたか

正解を与えることは、発見を奪う。
身体知の言語化において、気づきは外部から与えられるのではなく、問いによって内部から引き出される。
// CONTEXT — 背景
「腸で感じたこと」に対してLLMが「それはこういうことです」と正解を返す設計を最初に検討した。しかしこの設計は、目的と矛盾する。

身体知の言語化において重要なのは、記録者自身が「腑に落ちる」瞬間を経験することといえる。外部から与えられた言語は借り物であり、腸の回路を開かない。コーチングの原則「教えない・問う」は、ここでも成立すると考えている。
// DECISION — 決定
LLMのロールを「答える者」から「問う者」に転換している。

Socraticプロンプトは、記録の内容に対して問いを返すよう設計する。応答の最後に必ず「問い」が含まれ、記録者が次の腸の感知へと向かうことを促す。

評価基準も転換する。「正確さ」ではなく「共鳴するか」が成功指標になる。
// CONSEQUENCES — 帰結
+記録者自身の内省が深まる。腸の回路が開く
+「正解」がないため、hallucination benchmarkが無関係になる
+「共鳴するか」という感覚的評価が、唯一の成功指標として成立する
−「答えを求めている人」には物足りない体験になりうる
−問いの質がプロンプト設計の精度に強く依存する
ADR-003 採択済み

なぜlocalStorage端末完結にしたか

個人の腸の記録は、個人のものといえる。
集合知として継承するのではなく、個人のパターンにこそ価値がある。
// CONTEXT — 背景
身体感覚の記録はプライベートな内的体験といえる。サーバーに送信・保存するアーキテクチャは技術的に可能だが、このプロジェクトの哲学的立場と合わない。

「集合知として継承する」という言葉が一度検討されたが、正確ではなかった。データは個人のデバイスを離れない。個人の腸の記録に意味があるのは、それが「その人のパターン」であるためで、平均化・集合化した時点で失われるものがある。
// DECISION — 決定
全記録データをlocalStorageに保存する。バックエンドを持たない。

外部通信先はapi.anthropic.com（LLM推論）のみ。自由入力テキスト（note）はAPIに送信しない。APIには感覚ラベル・meaningタグ・統計値のみを送る。

プライバシーの最も正直な保証は「コードを見れば確認できる」ことといえる。MITライセンスで公開されているコードが、プライバシーポリシー宣言の代わりになる。
// CONSEQUENCES — 帰結
+プライバシーが構造的に保証される（コードで検証可能）
+バックエンド不要。GitHub Pages完結が可能
+「個人のパターン」という本来の価値に集中できる
−デバイスを変えると記録が消える（JSONエクスポートで対応）
−複数デバイス間の同期はできない
ADR-004 採択済み

gut-ragの実態——写像と変換

検索は起きていない。
ガードレール上を辿る写像と、LLMによる自然言語への変換が起きている。
// CONTEXT — 背景
プロジェクト名に「RAG」を含む。しかしRAGという言葉は一般に、ベクトル検索による文書取得を連想させる。gut-ragで実際に起きていることを正確に記述する必要があった。

曖昧なまま「RAGです」と言い続けることは、設計の誠実さに反すると考えている。
// DECISION — 決定
gut-ragの動作を2段階の変換として記述する：

① 写像（決定論的）
EMOJI_KNOWLEDGE_MAP / MEANING_KNOWLEDGE_MAP による写像。感覚ラベル・meaningタグ → 知識文書ID。これはADR-001で述べたガードレール上を辿ることに相当し、確率的要素はない。

② 変換（自然言語化）
取得した知識文書と経験統計をコンテキストとして、LLMが自然言語の問いに変換する。LLMはここから関与する。

LLMの役割は「設計された地図を読み上げる案内人」に近い。ガードレールの外に出て情報を自由に探す主体とは異なる。
// CONSEQUENCES — 帰結
+動作が完全に透明・予測可能。「なぜこの応答か」を設計レベルで追跡できる
+ハルシネーションのリスクが限定的（事実を述べるのではなく問いを返すため）
+「RAG」という名称の内実を正直に語れるようになった
−「RAG」という名称に技術的な厳密さがない（設計者の意図的な妥協）
−写像テーブルの拡張は手動作業が必要
ADR-005 採択済み

なぜgut-loggerとgut-socraticが別アプリなのか

記録する行為と、問いを受ける行為は、異なる身体状態を必要とする。
同一画面に置くことは、両者の質を下げると考えた。
// CONTEXT — 背景
初期設計では、記録とSocratic対話を単一画面に置くことも検討した。しかし実装を進める中で、「記録する」という行為と「問いを受け、内省する」という行為が、異なる注意の向き・身体状態を要求することが見えてきた。

記録は「今ここ」の感知を素早く捉える行為。Socratic対話は「少し距離を置いて問いに向き合う」行為。これらを同じ画面に置くことは、どちらの質も下げると判断した。
// DECISION — 決定
gut-loggerとgut-socraticを別HTMLファイルとして分離する。

記録はgut-loggerで完結し、gut-socraticへの遷移は任意とする。2つの画面の往復が、腸→脳→腸という螺旋を物理的に体現する形になる。
// CONSEQUENCES — 帰結
+記録時と内省時の「場」が分かれ、それぞれの質が上がる
+螺旋の往復（記録 → 問い → 新たな記録）が画面遷移として可視化される
+片方だけを使うことも自然にできる
−遷移にひと手間かかる
−2アプリ間の状態（gut_next_q）の受け渡しに設計コストがかかる
腸（感知） → gut-logger 記録 感覚・文脈・meaning タグ
→
meaning タグ → C案：知識文書を呼ぶ MEANING_KNOWLEDGE_MAP
→
Socratic 深掘り → 問いの向き（根/境/識） D案v2：gut_next_q に保存
→
gut-logger ナッジバナー → 腸（新たな感知） 48h 有効・✕で手動消去
// APPENDIX — 実装ノート

sw.js：外部オリジンの横取り問題

// BUG: TypeError: body is already used
旧 sw.js は外部オリジン（api.anthropic.com）の fetch も横取りしていた。
レスポンスを r.clone() + return r で二重消費 → API 呼び出しが無音で失敗。
修正（v6）
// 外部オリジンは即座にバイパス
if (url.origin !== self.location.origin) return;
この1行をfetchイベントの先頭に入れる。GitHub Pages + Anthropic API の構成では必須になる。

教訓：PWAのサービスワーカーは、外部API通信と共存する場合にオリジンチェックが不可欠。外部オリジンをバイパスしないと、APIレスポンスのbodyが消費済みになりエラーになる。
